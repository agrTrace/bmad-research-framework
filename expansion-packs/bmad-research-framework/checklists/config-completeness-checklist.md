# 项目配置完整性检查清单

## 使用说明

本检查清单用于验证自动生成的项目配置方案的完整性、合理性和可执行性。在用户确认配置之前，必须逐项检查以确保配置质量。

## 检查分类

### A. 项目分析完整性 ✓

#### A1. 项目基本信息
- [ ] **项目类型** - 已明确识别项目所属的学科类型
- [ ] **复杂度等级** - 准确评估为 high/medium/low 之一
- [ ] **研究周期** - 预估时间合理，符合项目规模
- [ ] **主要挑战** - 识别出核心技术和方法挑战

#### A2. 专业领域识别  
- [ ] **领域覆盖** - 涵盖项目所需的所有专业方向
- [ ] **领域关联** - 专业领域间存在合理的协作关系
- [ ] **优先级** - 核心领域标记为高优先级
- [ ] **领域边界** - 避免领域职责重叠或空白

### B. 专业智能体配置检查 ✓

#### B1. 配置数量合理性
- [ ] **总体数量** - 专业智能体总数适中（建议4-8个）
- [ ] **高优先级** - 高优先级专家数量占比30-50%
- [ ] **中优先级** - 中优先级专家数量占比30-40%  
- [ ] **低优先级** - 低优先级专家数量占比20-30%

#### B2. 专业配置质量
- [ ] **领域匹配** - 每个专业智能体的domain字段准确
- [ ] **知识库绑定** - knowledge_base字段对应正确的专业知识库
- [ ] **职责描述** - role字段具体明确，避免模糊表述
- [ ] **团队分配** - team_assignment合理分布，团队规模均衡

#### B3. 配置一致性
- [ ] **模板统一** - 所有配置都使用research-specialist-template
- [ ] **参数完整** - 每个配置包含所有必需参数
- [ ] **命名规范** - domain和role的命名遵循学术标准
- [ ] **无冲突** - 不同智能体的职责无重叠或冲突

### C. 同行评议专家配置检查 ✓

#### C1. 评议覆盖度
- [ ] **主要领域** - 项目主要专业领域都有对应评议专家
- [ ] **层次分布** - 包含senior和expert不同级别的评议专家
- [ ] **评议重点** - focus_area覆盖技术、方法、创新等关键维度
- [ ] **评议阶段** - 明确参与proposal/interim/final等评议阶段

#### C2. 评议专家质量
- [ ] **专业匹配** - 评议专家的domain与项目专业需求匹配
- [ ] **权威性** - expertise_level设置合理，确保评议权威性
- [ ] **独立性** - 评议专家之间相对独立，避免观点单一
- [ ] **平衡性** - 既有宏观把握又有细节关注的评议专家

### D. 团队结构合理性检查 ✓

#### D1. 团队组织
- [ ] **团队数量** - recommended_teams数量适中（建议2-4个团队）
- [ ] **团队规模** - 每团队专家数量合理（建议2-4人）
- [ ] **协调方式** - coordination_method选择恰当
- [ ] **沟通频率** - communication_frequency设置合理

#### D2. 分工协作
- [ ] **职责分工** - 团队间职责划分清晰，无重叠
- [ ] **协作接口** - 团队间协作关系明确定义
- [ ] **负载平衡** - 各团队工作量相对均衡
- [ ] **扩展性** - 团队结构支持后续调整和扩展

### E. 资源需求评估检查 ✓

#### E1. 资源配置
- [ ] **计算资源** - computational_resources需求评估合理
- [ ] **数据需求** - data_requirements明确具体
- [ ] **设备需求** - equipment_needs符合项目要求
- [ ] **外部协作** - external_collaboration需求识别准确

#### E2. 约束条件
- [ ] **时间约束** - timeline_constraints考虑充分
- [ ] **资源约束** - 资源需求在可获得范围内
- [ ] **技术约束** - 技术实现约束分析到位
- [ ] **人员约束** - 人员配置符合实际情况

### F. 质量保证机制检查 ✓

#### F1. 检查点设置
- [ ] **关键节点** - review_checkpoints覆盖项目关键里程碑
- [ ] **评估指标** - quality_metrics设计科学合理
- [ ] **频率适中** - 检查频率既保证质量又不影响效率
- [ ] **可操作性** - 检查点设置具备可操作性

#### F2. 风险控制
- [ ] **风险识别** - 主要项目风险识别全面
- [ ] **缓解策略** - risk_mitigation措施具体可行
- [ ] **改进机制** - continuous_improvement机制设计合理
- [ ] **应急预案** - 风险应对策略完备

## 配置优化建议

### 常见问题及修正

#### 问题1: 专业智能体配置过多
**现象**: 专业智能体总数超过8个
**影响**: 协调复杂度增加，效率降低
**修正**: 合并相近专业，突出核心需求

#### 问题2: 团队分配不均衡  
**现象**: 某些团队专家过多，某些团队人数不足
**影响**: 负载不均，协作困难
**修正**: 重新平衡团队分配，确保规模合理

#### 问题3: 评议覆盖不全面
**现象**: 某些关键专业领域缺少评议专家
**影响**: 评议质量下降，质量风险增加
**修正**: 补充缺失领域的评议专家

#### 问题4: 资源需求过高
**现象**: 配置的资源需求超出项目预算
**影响**: 项目可行性降低
**修正**: 优化配置，平衡需求与约束

## 检查流程

### Step 1: 自动检查
运行配置验证脚本，自动检查基本完整性和格式正确性

### Step 2: 人工审查  
使用本清单逐项检查配置内容的合理性和质量

### Step 3: 用户确认
向用户展示检查结果，获得确认或修改建议

### Step 4: 迭代优化
根据检查结果和用户反馈优化配置方案

## 验收标准

### 基本标准（必须满足）
- [ ] 所有A类检查项目通过
- [ ] 90%以上的B类检查项目通过
- [ ] 80%以上的C-F类检查项目通过

### 优秀标准（建议达成）
- [ ] 95%以上的所有检查项目通过
- [ ] 配置方案获得用户完全认可
- [ ] 预估项目成功概率大于80%
